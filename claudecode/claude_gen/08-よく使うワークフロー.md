# 08. よく使うワークフローとベストプラクティス

> **概要**: 日常タスクのプロンプトレシピと、Claude Code を最大限活かすためのベストプラクティスをまとめる。
> **参照元**: <https://code.claude.com/docs/ja/common-workflows> / <https://code.claude.com/docs/ja/best-practices>

---

## プロンプトレシピ

### 新しいコードベースの理解

```text
give me an overview of this codebase
explain the main architecture patterns used here
what are the key data models?
how is authentication handled?
find the files that handle user authentication
trace the login process from front-end to database
```

広い質問から始めて特定領域に絞る。プロジェクトの用語で質問する。

### バグを効率的に修正

```text
I'm seeing an error when I run npm test
suggest a few ways to fix the @ts-ignore in user.ts
update user.ts to add the null check you suggested
```

再現コマンド・スタックトレース・断続的か一貫的かを伝える。

### リファクタリング

```text
find deprecated API usage in our codebase
refactor utils.js to use ES2024 features while maintaining the same behavior
run tests for the refactored code
```

### テスト

```text
find functions in NotificationsService that are not covered by tests
add tests for the notification service
add test cases for edge conditions
run the new tests and fix any failures
```

Claude は既存のテストファイルのスタイル・フレームワークに合わせる。

### プルリクエスト作成

```text
summarize the changes I've made to the authentication module
create a pr
enhance the PR description with more context about the security improvements
```

`gh pr create` で作成するとセッションが PR にリンクされ、`claude --from-pr 123` で再開できる。

### ドキュメント

```text
find functions without proper JSDoc comments in the auth module
add JSDoc comments to the undocumented functions in auth.js
```

### 画像を使う

- Claude Code ウィンドウにドラッグ&ドロップ、`Ctrl+V` で貼り付け（`Cmd+V` は不可）、またはパスを渡す。
- エラー・UI デザイン・図のスクリーンショットをコンテキストに使える。

### ファイル/ディレクトリ参照（`@`）

```text
Explain the logic in @src/utils/auth.js
What's the structure of @src/components?
```

`@` はファイルの完全な内容を会話に含める。ディレクトリはファイルリストを表示。

## セッション管理系ワークフロー

### 会話を再開

```bash
claude --continue          # 現在のディレクトリで最新のセッション
claude --resume            # リストから選択
```

### worktree で並列セッション

```bash
claude --worktree feature-auth
```

各 worktree は独立したブランチのチェックアウト。編集が衝突しない。

### 編集前に計画する（Plan Mode）

```bash
claude --permission-mode plan
```

セッション中は `Shift+Tab` でも切替。承認まで編集しない。

### 調査をサブエージェントに委譲

```text
use a subagent to investigate how our auth system handles token refresh
```

別コンテキストで探索し要約だけ返す。メインコンテキストを汚さない（[12-サブエージェント.md](12-サブエージェント.md)）。

### スクリプトにパイプ

```bash
git log --oneline -20 | claude -p "summarize these recent commits"
```

## ベストプラクティス

ほとんどのベストプラクティスは1つの制約に基づく: **コンテキストウィンドウはすぐ満杯になり、満杯になるとパフォーマンスが下がる**。

### 1. 検証手段を与える

Claude は「完了したように見える」時に停止する。テスト・ビルド・スクリプト・スクショなど、合否を返すチェックを与えるとループが自動的に閉じる。

- ×「メールを検証する関数を実装して」→ ○「validateEmail を書く。user@example.com は true、invalid は false。実装後にテストを実行して」
- 症状でなく**根本原因**に対処させる（「エラーを抑制するな」）。
- 成功の主張でなく**証拠**（テスト出力・戻り値・スクショ）を求める。
- 厳格にゲートするなら `/goal` 条件、Stop フック、レビューサブエージェントを使う。

### 2. 探索 → 計画 → 実装 → コミット

1. **探索**: Plan Mode でファイルを読ませ質問に答えさせる。
2. **計画**: 詳細な実装計画を作らせる（`Ctrl+G` でエディタ編集）。
3. **実装**: Plan Mode を抜けてコーディング・検証。
4. **コミット**: 説明的メッセージでコミット・PR。

> スコープが明確な小修正（タイプミス、変数リネーム）は計画をスキップして直接依頼する。

### 3. 具体的なコンテキストを与える

- タスクをスコープする（どのファイル・どのシナリオ・テスト設定）。
- 情報源を指す（「ExecutionFactory の git 履歴を調べて要約して」）。
- 既存パターンを参照させる（「HotDogWidget.php が良い例。パターンに従って」）。
- `@` でファイル参照、画像貼り付け、`cat error.log | claude` でパイプ。

### 4. 環境を整える

- **効果的な CLAUDE.md**（[07-メモリ-CLAUDEmd.md](07-メモリ-CLAUDEmd.md)）: 「削除したら Claude が間違えるか？」を各行に問う。膨れた CLAUDE.md は指示が無視される。
- **権限を設定**（[13-権限とセキュリティ.md](13-権限とセキュリティ.md)）: auto mode / 許可リスト / サンドボックス。
- **CLI ツールを使わせる**: `gh`, `aws`, `gcloud` など。`Use 'foo --help' to learn...`。
- **MCP サーバー**（[09-MCP.md](09-MCP.md)）、**フック**（[10-フック.md](10-フック.md)）、**スキル/サブエージェント/プラグイン**（[11](11-スキルとスラッシュコマンド.md)/[12](12-サブエージェント.md)）。

### 5. 効果的にコミュニケーションする

- コードベースの質問はシニアエンジニアに聞くように。
- 大きな機能はまず Claude にインタビューさせる（`Interview me using the AskUserQuestion tool ... write a complete spec to SPEC.md`）。仕様完成後に新セッションで実装。

### 6. セッションを管理する

- **早期・頻繁に方向転換**: `Esc` で中断、`Esc Esc` / `/rewind` で巻き戻し、`"Undo that"`、`/clear` でリセット。
- 同じ問題を2回以上修正したら `/clear` して、学びを込めた具体的プロンプトで新規開始。
- **コンテキストを積極管理**: タスク間で `/clear`、`/compact <instructions>`、`/btw` でサイド質問。
- **チェックポイント**: 送信プロンプトごとにチェックポイントが作られる。`/rewind` で会話/コード/両方を復元（git の代替ではない）。

### 7. 自動化とスケール

- **非対話モード**: `claude -p "prompt"`（`--output-format json` / `stream-json --verbose`）。
- **複数セッション**: worktree、Writer/Reviewer パターン。
- **ファイル横断のファンアウト**: `for file in ...; do claude -p "..." --allowedTools "Edit,Bash(git commit *)"; done`。
- **auto mode で自律実行**: `claude --permission-mode auto -p "fix all lint errors"`。
- **敵対的レビュー**: 新コンテキストのサブエージェントに diff をレビューさせる（`/code-review` スキル）。

### よくある失敗パターン

- **キッチンシンクセッション**（無関係タスク混在）→ `/clear`。
- **何度も修正**→ 2回失敗したら `/clear` して良いプロンプトで再開。
- **過剰な CLAUDE.md**→ 容赦なく削除、フックに変換。
- **信頼して検証しない**→ 常に検証手段を与える。
- **無限探索**→ 調査を狭くスコープ、サブエージェントを使う。
