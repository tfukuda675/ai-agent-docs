# 13. 権限とセキュリティ

> **概要**: Claude Code の権限モデル（ルール構文・モード）、保護されたパス、サンドボックスをまとめる。
> **参照元**: <https://code.claude.com/docs/ja/permissions> / <https://code.claude.com/docs/ja/permission-modes> / <https://code.claude.com/docs/ja/sandboxing>

---

## 権限システムの基本

| ツールタイプ | 例 | 承認が必要か |
| --- | --- | --- |
| 読み取り専用 | ファイル読み取り、Grep | いいえ（作業/追加ディレクトリ内） |
| Bash コマンド | シェル実行 | はい（読み取り専用コマンドを除く） |
| ファイル変更 | Edit/Write | はい |

> 権限ルールは**モデルではなく Claude Code が実装**する。CLAUDE.md やプロンプトは Claude の意図を形作るが、許可範囲は変えない。

`/permissions`（`/allowed-tools`）でルールを表示・管理。

---

## 権限ルール構文

形式は `Tool` または `Tool(specifier)`。評価順は **deny → ask → allow**（最初の一致が決定。特異性は順序を変えない）。

- `Bash` — すべての Bash コマンド（deny ではツールをコンテキストから削除）
- `Bash(npm run build)` — 正確なコマンド
- `Bash(npm run *)` — プレフィックス一致（`*` の前のスペースが重要。`Bash(ls *)` は `ls -la` に一致、`lsof` に不一致）
- `Read(./.env)` / `Read(**/.env)` — ファイル読み取り（gitignore セマンティクス）
- `WebFetch(domain:example.com)` — ドメイン一致
- `mcp__puppeteer__*` — MCP サーバーのツール
- `Agent(Explore)` — サブエージェント

### settings.json の例

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git commit *)",
      "Read(./docs/**)"
    ],
    "deny": [
      "Bash(git push *)",
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)"
    ]
  }
}
```

### パスのアンカー（Read / Edit）

| パターン | 意味 |
| --- | --- |
| `//path` | ファイルシステムルートからの絶対パス |
| `~/path` | ホームディレクトリから |
| `/path` | 設定ソースからの相対（プロジェクト設定なら `<project root>/path`） |
| `path` / `./path` | 現在のディレクトリから |

> 単一先頭スラッシュ `/Users/...` は絶対パスではない（設定ソース基準）。絶対パスは `//Users/...`。

### 注意点

- 複合コマンド（`&&`, `||`, `;`, `|`, 改行）は各サブコマンドを個別に照合する必要がある。
- `curl http://github.com/ *` のような引数制約は脆弱。ネットワーク制限は `curl`/`wget` を deny し `WebFetch(domain:...)` を使う、または PreToolUse フックで検証する。
- deny ルールは組み込みファイルツールと認識される Bash（`cat`, `head` 等）に適用されるが、Python/Node スクリプトが直接開くファイルには適用されない → OS レベルの強制は[サンドボックス](#サンドボックス)。

## 権限モード

| モード | 確認なしで実行 | 用途 |
| --- | --- | --- |
| `default`（Manual） | 読み取りのみ | 開始・機密作業 |
| `acceptEdits` | 読み取り+ファイル編集+一般的な fs コマンド（作業ディレクトリ内） | レビュー中の反復 |
| `plan` | 読み取りのみ（編集しない） | 探索・変更前 |
| `auto` | すべて（分類器の背景チェック付き） | 長時間タスク |
| `dontAsk` | 事前承認済みツールのみ（他は自動拒否） | ロックダウン CI |
| `bypassPermissions` | すべて | 隔離コンテナ/VM のみ |

### 切替

- **セッション中**: `Shift+Tab` で `default → acceptEdits → plan` を循環（`auto` は条件を満たすと表示）。
- **起動時**: `claude --permission-mode plan`。
- **既定**: `settings.json` の `permissions.defaultMode`。

```json
{ "permissions": { "defaultMode": "acceptEdits" } }
```

### auto モード

別の分類器モデルがアクションを実行前にレビューし、スコープ逸脱・未知インフラ・敵対的コンテンツ駆動のアクションをブロックする。

- Anthropic API では Opus 4.6+ / Sonnet 4.6+。Team/Enterprise では所有者が有効化する必要あり。
- 会話で述べた境界（「プッシュしないで」）もブロック信号として扱う。
- 3回連続 or 合計20回ブロックで一時停止しプロンプトに戻る（`-p` 非対話モードでは中止）。
- 既定でブロックする例: `curl | bash`、機密データの外部送信、本番デプロイ、フォースプッシュ、`git reset --hard`、`terraform destroy` など。
- `claude auto-mode defaults` で全ルールを確認。信頼インフラは `autoMode.environment` で追加。

### bypassPermissions の警告

権限プロンプトと安全チェックを無効化する。**Claude Code がホストに損害を与えられない隔離環境（コンテナ/VM/ネットなし dev container）でのみ使う**。プロンプトインジェクションを防がない。Linux/macOS で root/sudo 実行時は起動を拒否する。`--dangerously-skip-permissions` と同等。組織は `permissions.disableBypassPermissionsMode: "disable"` でブロック可能。

## 保護されたパス

`bypassPermissions` 以外の全モードで自動承認されない書き込み先（リポジトリ状態・Claude 設定の破損防止）:

- ディレクトリ: `.git`, `.config/git`, `.vscode`, `.idea`, `.husky`, `.cargo`, `.devcontainer`, `.yarn`, `.mvn`, `.claude`（`.claude/worktrees` を除く）
- ファイル: `.gitconfig`, `.bashrc`/`.zshrc`/`.profile`/`.envrc` 等、`.npmrc`/`.yarnrc` 等、`.mcp.json`, `.claude.json` など

`permissions.allow` でも保護パスへの書き込みは事前承認されない。

---

## 作業ディレクトリ

既定では起動ディレクトリ内のファイルにアクセス可能。拡張:

- 起動時: `--add-dir <path>`
- セッション中: `/add-dir`
- 永続: `settings.json` の `permissions.additionalDirectories`
- 主要作業ディレクトリの移動: `/cd`（`Cd()` ルールで制限可能）

---

## サンドボックス

サンドボックスは **Bash ツールのファイルシステム/ネットワークアクセスを OS レベルで制限**する（Bash とその子プロセスに適用）。権限（ツール/ファイル/ドメイン制御）と補完的で、両方使うと多層防御になる。

- `/sandbox` で切替（対応プラットフォームのみ）。管理設定 `sandbox.enabled` で強制。
- ファイルシステム制限は `sandbox.filesystem` + Read/Edit deny ルールの和集合。
- ネットワーク制限は `sandbox.network` の `allowedDomains`/`deniedDomains` + WebFetch 権限の組み合わせ。
- `autoAllowBashIfSandboxed: true`（既定）だと、サンドボックス内 Bash はプロンプトなしで実行される（deny ルールと `rm -rf /` 等の回路遮断器は残る）。
- 詳細: <https://code.claude.com/docs/ja/sandboxing>

---

## 設定の優先順位（再掲）

1. 管理設定（オーバーライド不可）
2. コマンドライン引数
3. ローカル（`.claude/settings.local.json`）
4. プロジェクト（`.claude/settings.json`）
5. ユーザー（`~/.claude/settings.json`）

いずれかのレベルで deny されると、他レベルで allow できない。管理設定でユーザー/プロジェクトが上書きできない組織ポリシーを配布できる（`permissions.deny`、`sandbox.enabled`、`forceLoginMethod` など）。

---

## セキュリティ上の注意

- MCP サーバーや外部コンテンツはプロンプトインジェクションのリスク（[09-MCP.md](09-MCP.md)）。信頼できるサーバーのみ接続する。
- プロジェクトの `.claude/settings.json` の `allow` ルール・追加ディレクトリは、ワークスペーストラストダイアログ受諾後にのみ適用される（`deny`/`ask` は制限のみなので常に適用）。
- 必ず実行/ブロックさせたいルールは PreToolUse フックで（[10-フック.md](10-フック.md)）。フックの deny は `bypassPermissions` でもブロックする。
