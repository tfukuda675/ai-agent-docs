# RAG 設計ガイド（.md 化済みドキュメントから高精度 RAG を作る）

> **概要**: pptx 由来で .md 化したドキュメント（本リポジトリの `docs_from_agent/` のような資料）を対象に、精度の高い RAG を実装するための設計方針をまとめる。
> **対象**: `claude-code/` `opencode/` などのトピック別 Markdown 群を検索・回答に使いたい場合。

---

## 結論（おすすめの優先順位）

**まず「コーパスの規模」で設計方針が決まる。** 本リポジトリの資料は全体で数百 KB・数十ファイル程度と小さいので、いきなりベクトル RAG に行く前に検討すべき選択肢がある。

| 規模 | 推奨アプローチ | 理由 |
| --- | --- | --- |
| 小（〜数百 KB／〜数十ファイル） | **全文投入 + プロンプトキャッシュ**（＝ RAG しない / CAG） | 検索の取りこぼしがゼロ。実装が最小。精度が最も高い |
| 中（数 MB〜） | **構造対応チャンク + ハイブリッド検索 + リランク**（+ Contextual Retrieval） | ベクトルだけだと固有語（`--permission-mode` 等）を取り逃す |
| 大（多数のプロダクト・版） | 上記 + **メタデータ・ルーティング** | 「Claude Code か opencode か」等で先に絞る |

今回の `docs_from_agent` サイズなら、**まず一番上の方式**を強く推奨する。

---

## 1. 小コーパスなら「全文 + プロンプトキャッシュ」

このドキュメント群は、質問対象ごと（claude-code / opencode）に**丸ごとコンテキストに入る**サイズ。

- 該当ツールの全 `.md` を連結してシステムプロンプト末尾に置き、`cache_control: {"type": "ephemeral"}` を付ける。
- 2 回目以降のクエリは**キャッシュ読み出し（約 0.1 倍のコスト）**になり、レイテンシもコストも激減する。
- 検索段の取りこぼし・チャンク分割ミスが原理的に発生しない。

```python
# Anthropic SDK（Python）。モデルは claude-opus-4-8
resp = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=16000,
    system=[{
        "type": "text",
        "text": FULL_DOCS_TEXT,                 # claude-code/ 全 .md を連結
        "cache_control": {"type": "ephemeral"}, # ここまでをキャッシュ
    }],
    messages=[{"role": "user", "content": user_question}],
)
```

> キャッシュは**前方一致**。システムプロンプト先頭に日付や乱数を入れると無効化される。可変部分（質問）は必ず後ろに置く。

## 2. 本格 RAG にするなら（コーパスが増える前提）

ベクトル RAG を組む場合、**この種の技術ドキュメントで精度を上げる勘所**は以下。

### (a) チャンク分割 — 構造対応（固定長で切らない）

今回の .md は見出し（H2/H3）・表・コードブロックが整っている。これを活かす。

- **Markdown ヘッダー単位で分割**（`##`/`###` の節を 1 チャンク）。固定トークン分割は使わない。
- **表とコードブロックは絶対に分断しない**（`settings.json` 例やコマンド表が途中で切れると致命的）。
- 各チャンクの先頭に**見出しパス**を付与（例: `claude-code > 13-権限とセキュリティ > 権限モード`）。

### (b) メタデータ — 既存の構造を再利用

`document-index.md` のキーワードタグや章立てがそのままメタデータになる。

- `tool`（claude-code / opencode）、`topic`（認証 / MCP / 権限…）、`heading_path`、`source_url`。
- 検索時に `tool` で**事前フィルタ**すれば、ツールをまたいだ誤ヒットが消える。

### (c) Contextual Retrieval（精度が大きく上がる）

各チャンクの埋め込み前に「このチャンクが文書全体のどこに位置するか」の短い文脈文を先頭に付けてから埋め込む手法。単独では意味が薄いチャンク（表の断片など）の検索精度が大幅に改善する。疎検索側（BM25）にも同じ文脈文を含めると効果的。

### (d) ハイブリッド検索 — 技術ドキュメントの肝

このドキュメントは `--permission-mode`、`autoUpdatesChannel`、`opencode run` のような**完全一致で当てたい固有語**が多い。

- **密ベクトル（意味検索）＋ 疎検索（BM25/キーワード）**を併用する。
- さらに**リランカー**で上位 k を再順位付けする（Cohere Rerank / LLM リランク等）。
- 埋め込みモデルは**日本語対応（多言語）**のものを選ぶ。Anthropic 自体は埋め込みを出しておらず、Voyage AI 等が推奨される。

### (e) 生成 — Claude で回答＋出典明示

- 生成モデルは **`claude-opus-4-8`**（最新・最上位）。取得チャンクをコンテキストに入れて回答させる。
- **出典（ファイル名 + 見出し）を必ず引用**させる。ドキュメントに「執筆時点の仕様／バージョン注記」がある性質上、これは重要。
- コンテキストに無いことは「記載なし」と言わせる（ハルシネーション抑制）。

### (f) 評価

- 実際に来そうな質問を 20〜50 問用意し、「正しいチャンクを retrieval できたか」と「回答が正しいか」を**分けて計測**する。
- チャンクサイズ・ハイブリッドの重み・top-k を、この評価セットで調整する。

## この資料特有の注意点（pptx 由来 → .md）

- 今回の .md は整形済みでクリーンだが、**一般に pptx→md 変換は表・レイアウト・図中テキストが崩れやすい**。RAG 前に「表が壊れていないか」「図の情報が落ちていないか」を必ず点検する（崩れたチャンクはそのまま検索ノイズになる）。
- **バージョン注記**（「v2.1.x 以降」「執筆時点 2026-07」）がチャンク単位で失われないよう、メタデータか本文に残す。

---

## まとめ

- 今の規模なら **「該当ツールの .md 全文＋プロンプトキャッシュ」が最も高精度・低コスト**。まずこれで十分。
- 将来コーパスが増えたら、**構造対応チャンク → メタデータフィルタ → ハイブリッド検索＋リランク → Contextual Retrieval** の順で足していくのが、この種の技術ドキュメントでは効く。
- 生成は `claude-opus-4-8`＋出典明示。

---

## パイプライン全体像（本格 RAG）

```
[.md 群]
  → 構造対応チャンク分割（見出し単位・表/コード保持）
  → 見出しパス付与 + メタデータ抽出（tool / topic / source_url）
  → Contextual Retrieval 用の文脈文を前置
  → 埋め込み（多言語モデル） + BM25 インデックス
--------------------------------------------------
[クエリ]
  → メタデータ事前フィルタ（tool 等）
  → 密ベクトル検索 + BM25（ハイブリッド）
  → リランク（上位 k）
  → Claude opus-4-8 で回答生成（出典明示・未記載は「記載なし」）
  → 評価セットで retrieval 精度と回答精度を分けて計測
```
